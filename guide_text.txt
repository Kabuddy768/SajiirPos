1. Project Identity














Sajiir POS is a multi-tenant, offline-capable Point of Sale system built specifically for Kenyan retail businesses â€” supermarkets, general trade, and FMCG merchants. It is built with Django and is deployed as a single codebase that serves multiple client businesses, each isolated in their own PostgreSQL schema.


























1.1 Core Mission



















A cashier can scan items, accept payment (Cash, M-Pesa STK Push, or Card), and print a KRA-compliant receipt in under 30 seconds



















A manager can write off expired or spoiled stock with a full audit trail



















A business owner can see consolidated revenue across all branches in one dashboard



















Sales continue even when the internet goes down (offline-first via Service Worker + IndexedDB)



















Every sale is submitted to KRA eTIMS for VAT compliance


























1.2 Target Users



















Kenyan supermarkets and retail shops (single or multi-branch)



















VAT-registered businesses required to use KRA eTIMS



















Cashiers using desktop browsers (not mobile, not a native app)


























1.3 What Makes This Different



















Stock is a ledger, not a counter â€” every movement has a reason, a who, and a when



















Carton-to-unit conversion built in â€” buy 10 cartons of 24 units, stock shows 240 units



















M-Pesa STK Push is native, not bolted on



















Offline mode is a first-class feature, not an afterthought



















KRA eTIMS compliance is baked in from day one



















Per-branch stock â€” a product&apos;s stock at Branch A is independent of Branch B

7. Absolute Rules â€” Never Break These














These rules exist because violating them causes data corruption, not just bugs. They are hard constraints, not guidelines.



























































RULE 1





























Never hard-delete products, sales, customers, or suppliers. Use is_active=False. Historical sales must always resolve their original product name and price.






























































RULE 2





























Stock is a ledger. BranchStock.quantity is maintained only by the post_save signal on StockMovement. Never write to BranchStock directly from a view, serializer, or service. Always call StockService.adjust().






























































RULE 3





























All sale completion logic lives in SaleService.complete() inside transaction.atomic(). Views call this method â€” they never create Sale, SaleItem, Payment, or StockMovement directly.






























































RULE 4





























Offline sync is idempotent. Check offline_uuid against the Sale table before calling SaleService.complete(). Duplicates must return already_synced silently, never raise an error.






























































RULE 5





























The M-Pesa callback view has no authentication. It must always return HTTP 200 with {&quot;ResultCode&quot;: 0}. Non-200 causes Daraja to retry for hours.






























































RULE 6





























GRNItem carton conversion happens in GRNItem.save() â€” not in views or services. quantity_sale_units = quantity_purchase_units * product.units_per_purchase.






























































RULE 7





























Celery tasks that query the database must call schema_context(schema_name) first. Without it, the ORM hits the public schema and finds nothing.






























































RULE 8





























AuditLog is append-only. Never update or delete AuditLog rows. Write via AuditMixin or AuditService.log() â€” never from views directly.

9. Key Model Relationships














These relationships must be understood before touching any business logic.


























Stock flow



















Product has no stock field â€” stock lives in BranchStock (per branch)



















Every stock change creates a StockMovement row (append-only)



















The post_save signal on StockMovement updates BranchStock using assignment (=), not += 



















StockService.adjust() is the ONLY permitted way to create a StockMovement


























Sale flow



















A Sale belongs to a CashSession (cashier&apos;s shift)



















A Sale has many SaleItems (price snapshot â€” never changes after creation)



















A Sale has many Payments (Cash + M-Pesa + Card on the same sale is valid)

12. Context and Decisions Made














Important decisions already locked in â€” do not revisit these without a strong reason.




























UI framework: Tailwind CSS (not Bootstrap 5 as originally requested). All templates use Tailwind consistently. Switching would require rewriting every template for no functional gain.



















Database: PostgreSQL with django-tenants schema-per-tenant isolation. SQLite is not supported.



















Auth: Email-based (not username). JWT for the API, Django session auth for template views.



















Offline mode: Service Worker + IndexedDB. Every sale gets a browser-generated UUID to prevent duplicates on sync.



















Payments: Cash confirmed immediately. M-Pesa waits for Daraja callback before marking confirmed. Card is manual reference entry.



















Stock ledger: BranchStock is a summary, StockMovement is the truth. This pattern must not change.



















KRA eTIMS: Every sale must be signed. The ETIMSClient is a stub â€” real KRA API credentials needed for production.



















Carton conversion: Happens in GRNItem.save(). The view/form collects purchase-unit quantities; the model computes sale-unit quantities automatically.



















Sale number format: {BRANCH_CODE}-{YYYYMMDD}-{SEQUENCE} e.g. NBR-20260427-0001



















Return approval threshold: KES 5,000 â€” returns above this require manager approval (field approved_by on Return model).



















Loyalty points: 1 point per KES 100 spent, rounded down. No expiry for now.


























Design decisions that must NOT be changed


















































LOCKED





























BranchStock is never written to directly. Only the StockMovement signal updates it. If you find code that does BranchStock.objects.filter(...).update(quantity=...) or branch_stock.quantity = X; branch_stock.save() outside the signal â€” that is a bug.






























































LOCKED





























SaleService.complete() is the single point of entry for creating a sale. If you find a view that creates a Sale object directly, that is a bug.






























































LOCKED





























The M-Pesa callback view has no authentication classes. This is intentional. Safaricom cannot authenticate. Protect it with HMAC verification instead (optional, see MPESA_CALLBACK_SECRET env var).
